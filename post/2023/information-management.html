Information Management

First published 2021–11–12; substantive revision 2023–01–16 ~1600 words

Personal information management (Pim) can be helpful to—via the intention of repurposing the collected semantic information—create something worthwhile in the future.

But knowledge management certainly also has a negative connotation due to its misguided usage in the milieu of status-seeking productivity-optimization and self quantification; as an end in itself.[1] A tendency of information hoarding may also be a sign of self-justification of excessive distraction seeking via information consumption. Nevertheless, the notion of writing to forget, or let go of thoughts and ideas, alone could substantiate such practices as practical release valve as long as one is aware of the rationale behind them.

Information Science

As an individual’s personal space of information (Psi), now increasingly shifts from the physical to the digital, and with growing importance of human–computer interaction (Hci), digital Pim[2] becomes the main resource for information keeping and Pkm. As knowledge is only indirectly manageable,[3] Pkm is regarded a subset of Pim and should, for my taste, also utilize spaced repetition to facilitate learning as its primary objective. With that, Pim also involves the cognitive science fields of problem solving and decision making.

Within information science, the field of Pim is unique in its self-directed nature, curation and retrieval of items is done by the same entity. The user-subjective approach describes the need for flexibility of a given system to adapt to its user, enabling incorporation of various information fragments into a single structure. Primacy effect[4] and the active categorization required are most likely responsible for the preference of Pim-system design into folder trees and its traversal via navigation, in contrast to search functions, tagging, or relationship graphs. Information fragmentation is a problem exaggerated by the ever increasing amount of information items created and amount of external services used, thus the ability to export user data has to be a prerequisite. Digitization of non born-digital assets can be performed to consolidate the Psi.

File Management

File Management is about designing systems for information and data preservation. Data preservation is concerned with present time data loss prevention and thereby protection of privacy, but also goes beyond that by anticipating usablility, integrity and other potential data object changes in the future. Data loss prevention is a prerequisite for further preservation and is most basally implemented with a backup scheme.1 Future-proofing is ensured by using established standards for file management structures and file formats to make renderability more likely and future retrieval more convenient. Further, time-tested and widely used Foss often has a higher chance of maintained development in comparison to proprietary software alternatives.

Implementing hierarchical folder-tree structures, analogous to the use of universal decimal classification (Udc)[5] and Ddc in library classification demands compromises in content allocation edge cases, but rewards with reduced infrastructure maintainance work and clarity. If decimal classification is used to name certain classes of files that can actually benefit from it, it has to be considered that for most file managers, numerical sorting is the default, meaning 5 < 22 < 100. This has to be disabled, as the numbers in this scheme are used as if they were decimals, hence .100 < .22 < .5. Grouping in decimal classification follows defined categories, the need for interlinking notes, as is required with nonhierarchical relationship graphs, wikis or tagging systems, is limited. Traversing the distance between two related nodes within the data structure should minimize involvement of ancestors.

There should be content type dependent file naming conventions. For sorting purposes, the most relevant metadata is stated first. While books could be named via UDC - Title - Author - Year, others types may be sorted by initializing the filename with Iso–8601 (2021-11-12). Choosing the correct date to associate with a file can be tricky in some cases, for example consider a video shot in 2000, published in 2012 and introduced to the Psi in 2020. Plain hierarchical folders are not the best fit for all applications, they can however be expanded to serve as storage location for, hence working in conjunction with, tag editor generated databases to the benefit of image categorization.

Studies advocate for the GrayArea concept [6], [7]. A GrayArea is implemented as a gray area in each folder, enabling relevance classification of data into a third state between keep and delete by graying files out. Personally, the concept of GrayArea seems like a bad one, because you simply shift the issue around. I nonetheless use an auxiliary-folder named !tmp, that can be used as an incorporation buffer for files violating naming conventions, with the distinction that its temporal aspect is eponymous. With regular backups, and versioning being taken care of, the caution threshold for removing data is lowered to break the tendency of overkeeping.

Notes and Annotations

Since plain text is the most flexible, writing in a lightweight markup language, in this case Markdeep-js markup syntax, is the best option. Markdeep allows writing browser-accessible html documents in an extended markdown syntax, with the caveat that browsers do not support XMLHttpRequest to the local file system or shell.openPath(), so files from the local file system cannot be opened by the browser with a local application 2. Over time, however, simply reading and writing these files in a text editor proved to be the most practical, making its ability to generate html somewhat obsolete. Even for personal notes, having a certain standard in relation to style and reference management will ensure they can contribute to potentially distributable assets 3.

Reference Management

Output generation like non-fiction writing is build upon a foundation of curating and validation checking sources. A structured information input, as shown in the graph, facilitates this. One has to be aware that this can make consumption seem more productive and worthwhile than it probably is, just because you create a record of it does not mean that it was worth your time. The file naming conventions for different content types from file management result in the ability to convert a list of filenames with regular expressions to a desired metadata format. XML is the lingua franca for serialization of information, other formats suffice when the given file’s conversion to a pandas dataframe or manipulation with regular expressions is straightforward, as is the case for .csv and .bib files. Thusly, tools to be utilized are required to offer those export schemes. The biblatex format is a good option in combination with the Jabref reference manager and the pandoc document conversion software used to generate various types of output documents from the notes and references. In libraries, the subject cataloging facilitated by decimal classification is usually augmented with controlled vocabulary tagging, e.g. Library of Congress Subject Headings or GND (Gemeinsame Normdatei). While this is helpful for information retrieval, it has a high maintenance cost as biblatex is far from ideal for this application. Defining a custom bibliography style for custom entry-types and -fields is very involved, but would be necessary to keep the records in line with other authority files (wikidata, GND). Bibliographic records can be hierarchically structured into work, expression, manifestation and item (WEMI). Biblatex is designed to cover only the manifestation subset.

On the plus side, in mp3tag, the audio file metadata tag editor, a custom export filter can be defined to export albums as entries to biblatex format.

In Jabref too, custom export templates can be created to export data as desired. These two cover most content types adequately with the notable exception of images. digiKam is a FOSS image tag editor to organize images, what it lacks is a way to export a spreadsheet of image metadata from the populated database. This is useful if storing tags within the file, or doubling the file count by using XMP sidecar files is undesired. By default, digiKam stores metadata in the digikam4.db sqlite database. There are no custom fields or any kind of build-in metadata export in digiKam, thus it is not a perfect solution. The following generates a table with filenames and associated tags.

-- !!! Generate a table with filename and corresponding tags from digikam4.db
-- this table has the tagid and the corresponding tag name
CREATE TABLE IF NOT EXISTS t1 AS 
SELECT tagid, name 
FROM ImageTags 
INNER JOIN tags ON tags.id = tagid 
GROUP BY tagid;

-- this table replaced imagetags.tagid with the tag name
CREATE TABLE IF NOT EXISTS t2 AS 
SELECT imageid, name 
FROM ImageTags 
INNER JOIN t1 ON ImageTags.tagid = t1.tagid;

-- we concat the tag names that belong to an image id, and show the filename for each image id
SELECT images.name AS 'FileName', GROUP_CONCAT(t2.name, '; ') AS 'Tags'
FROM t2 
INNER JOIN images ON images.id = imageid 
GROUP BY imageid;

-- we remove the temporary tables 
DROP TABLE t1;
DROP TABLE t2;
